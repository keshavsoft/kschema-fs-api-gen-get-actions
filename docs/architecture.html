Architecture

How the JSON-driven dynamic command loading operates inside the CLI core.

Dynamic Execution Flow

KSchema uses an ESM dynamic import strategy, which avoids bloating runtime configurations and eliminates maintaining hardcoded import maps.

CLI input parameters
↓
parseInput.js
↓
resolveCommand.js
↓
loadCommand.js fetches configuration from actions.json
↓
dynamic import() on target task action
↓
Scaffolding template generation

Configuration Structure (`actions.json`)

The CLI checks the actions metadata file for mapping terminal parameters. Here is a configuration snippet:

json
[ { "cmd": "showAll", "file": "showAll", "exportFile": "ShowAll", "group": "GetMethods", "description": "Generate ShowAll GET action" } ]

Dynamic vs Static Architecture

Older Architecture

❌ Manual static imports
❌ Hardcoded command mappings
❌ High maintenance code paths
❌ Slower scaling to new actions

Modern Dynamic Loader

✅ Dynamic ESM imports
✅ Single configuration source (`actions.json`)
✅ Zero code churn for new actions
✅ Clean, decoupled codebase
